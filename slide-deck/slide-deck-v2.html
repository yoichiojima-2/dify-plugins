04 — ツール構成
13ツール、すべて単一プラグインに統合
売上分析・ダッシュボード
sales_analytics / dashboard_generator / dashboard_template — KPI集計からインラインHTML生成まで
シフト管理
shift_manager / shift_optimizer / shift_table_generator — CRUD・最適化・表生成
需要予測・発注
hourly_weather + demand_forecast (RandomForest) + order_optimizer — 天気連動の発注数算出
在庫管理
inventory_manager — 在庫レベル管理・消費期限アラート・不足検知
LINE連絡・商品情報
line_composer / lawson_items — テンプレート連絡文生成・商品カタログ検索
共通基盤
datetime_utils / db_utils / data_loader — JST変換・DuckDB接続・JSONキャッシュ
外部API依存は天気（Open-Meteo、無料）のみ。データはすべてインメモリ（DuckDB）で処理。Dify Cloudでもセルフホストでも動く。

08 — 設計思想
なぜこう作ったか
チャットフローではなくエージェント
店長の依頼は毎回違う。「シフト作って」と「天気見て発注して」と「田中さんが休みだからLINEも」では使うツールが全部違う。固定フローでは対応しきれないのでエージェントモードを採用した
13ツールを1プラグインに
依存関係を増やしたくなかった。別々のプラグインにすると管理が煩雑になるし、ツール間でデータ共有ができない。DuckDBの接続もJSONのキャッシュもプラグイン内で完結させた
インメモリDB（DuckDB）
Dify Cloudは読み取り専用ファイルシステム。外部DBに依存すると環境構築のハードルが上がる。DuckDBならプラグイン起動時にメモリ上にスキーマを作るだけで動く
外部API依存の最小化
APIキーの設定が必要になると使ってもらえない。天気だけはリアルタイム性が必要なのでOpen-Meteo（無料・キー不要）を使い、他のデータはすべて内蔵した
ユニットテスト107件。ツールごとにテストを書いているので、機能追加やリファクタリングのときに壊れたらすぐわかる。

09 — まとめ
からあげ店長クン
何をするか
コンビニ店長の日常業務——売上分析、シフト管理、需要予測、在庫管理、連絡文作成——をDifyエージェントが自然言語の指示で実行する
なぜ価値があるか
ベテランの暗黙知に頼っていた発注判断やオペレーションを、誰が店長でも一定の品質で回せるようにする。業務の平準化と高度化を同時に狙う
技術構成
Dify Agent Mode + 13ツール
DuckDB（インメモリDB）
RandomForest（需要予測）
Open-Meteo（天気API）
インラインHTML描画
設計方針
外部API依存の最小化
Cloud / セルフホスト両対応
ファイルシステム書き込み不要
