03 — 技術構成
エージェント + 13ツール + インメモリDB + MLモデル
処理の流れ
1
自然言語入力— ユーザーがチャットで依頼
2
ツール選択（LLM推論）— 13ツールから必要なものを判断
3
実行— DuckDB / RandomForest / Open-Meteo API
4
成果物出力— HTMLダッシュボード・シフト表・LINE文
外部API依存は天気（Open-Meteo、無料）のみ。それ以外のデータはすべてプラグイン内で完結。Dify Cloudでもセルフホストでも動作する。

04 — 機能一覧
13ツールの構成
売上分析・ダッシュボード
sales_analytics, dashboard_generator, dashboard_template の3ツールでKPI集計からHTML生成まで
シフト管理
shift_manager, shift_optimizer, shift_table_generator でCRUD・最適化・表生成を処理
需要予測・発注
hourly_weather + demand_forecast (RandomForest) + order_optimizer で天気連動の発注数算出
在庫管理
inventory_manager で在庫レベル管理・消費期限アラート・在庫不足検知
LINE連絡文・商品情報
line_composer でテンプレート基盤の連絡文生成。lawson_items で商品カタログ検索
共通ユーティリティ
datetime_utils (JST変換・曜日)、db_utils (DuckDB接続管理)、data_loader (JSONキャッシュ)
すべてインメモリで動作（DuckDB）。ファイルシステム書き込み不要のため、Dify Cloudの読み取り専用環境でも動く。

06 — デモ②
天気連動の発注最適化
「天気を見て明日の発注を最適化して」
ツール呼び出し（4ツール連鎖）
hourly_weather → 明日の天気・気温を取得
demand_forecast → RandomForestで商品別需要予測
inventory_manager → 現在庫と発注点を確認
order_optimizer → 推奨発注数を算出
4つのツールをLLMが自律的に連鎖実行。固定のワークフローではない。
実行結果のスクリーンショット
天気→需要予測→在庫照合→発注推奨の出力

08 — 評価軸に対する整理
このプラグインの位置づけ
評価軸 本プラグインの対応
新規性 チャットフローやRAGではなく、エージェントモードでツール呼び出しを動的に組み立てる。1つの自然言語入力に対して複数ツールを連鎖実行する構成は、Difyアプリとしては一般的でない
独自性 コンビニ店長業務に完全特化。シフト管理・需要予測（ML）・在庫管理・LINE連絡文生成を単一プラグインに統合。外部SaaS依存なし
採算性 対象市場は国内コンビニ約56,000店舗。1店舗あたり年間廃棄468万円という課題規模がある。発注精度の改善は直接的なロス削減につながる領域
パフォーマンス DuckDBインメモリ処理で高速応答。MLモデル（RandomForest）はプラグイン内蔵で推論が軽い。外部API呼び出しは天気のみ
使いやすさ 自然言語で操作。専門知識やUI操作は不要。GitHubからURLを貼るだけでインストール完了
